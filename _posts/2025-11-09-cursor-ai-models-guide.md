## 用語解説: コンテキストウィンドウとは？

### トークンとは
**トークン**は、AIモデルがテキストを処理する際の最小単位です。

- 英語の場合: 1単語 ≈ 1〜2トークン
- 日本語の場合: 1文字 ≈ 1〜2トークン
- 例: "Hello World" ≈ 2トークン
- 例: "こんにちは" ≈ 5〜10トークン（モデルによって異なる）

### コンテキストウィンドウとは
**コンテキストウィンドウ**は、AIモデルが一度に「記憶」して処理できる情報量の上限です。これは、**AIの「作業机の広さ」**に例えることができます。

- 机が小さい（8K）= 少しの資料しか広げられない
- 机が大きい（2M）= 大量の資料を同時に広げて全体を見渡せる

### サイズ感の目安

| コンテキストサイズ | 日本語換算 | 具体例 |
|------------------|-----------|--------|
| 8K トークン | 約4,000文字 | 中程度の関数1つ、短いファイル |
| 16K トークン | 約8,000文字 | 小規模なファイル1〜2個 |
| 128K トークン | 約64,000文字 | 書籍の1章分、中規模ファイル5〜10個 |
| 200K トークン | 約100,000文字 | 中規模プロジェクトの主要部分 |
| 1M トークン | 約500,000文字 | 長編小説1冊分、中規模プロジェクト全体 |
| 2M トークン | 約1,000,000文字 | 百科事典1冊分、大規模プロジェクト |

### なぜ重要なのか

#### ❌ コンテキストウィンドウが小さい場合
- 全体像が把握できず、的外れな提案になる可能性
- ファイル間の依存関係を理解できない
- 複数ファイルにまたがる変更が困難

#### ✅ コンテキストウィンドウが大きい場合
- プロジェクト全体を一度に読み込める
- ファイル間の関係性を理解した提案が可能
- 一貫性のあるリファクタリングが実現
- プロジェクト全体に影響する変更も安心

### 実践例: 認証システムの移行

**シナリオ**: 既存の認証システムを OAuth2 に移行したい

**小さいウィンドウ（8K）の場合**:
```
「auth.jsファイルだけを見て修正提案」
→ このファイルは他の20ファイルから呼ばれている
→ 修正すると他の部分が壊れる可能性が見えない
→ 影響範囲を把握できない
```

**大きいウィンドウ（2M）の場合**:
```
「プロジェクト全体の50ファイルを同時に読み込み」
→ auth.jsを使っている全ての場所を特定
→ 全ファイルを一貫性を保って修正
→ 影響範囲を完全に把握した提案が可能
→ テストファイルも含めて包括的な変更
```

|--------------|--------------|-----------|------|

### モデル別のコスト特性

各AIモデルはAPIコストが異なるため、同じプランでも選択するモデルによって利用可能な回数が変わります。

**コストが高いモデル（高精度・高性能）**:
- Claude 3 Opus
- GPT-4
- Gemini 2.5 Pro（大量のコンテキスト使用時）

**コストが中程度のモデル（バランス型）**:
- Claude 3.5 Sonnet
- GPT-4 Turbo
- GPT-4.1
- Gemini 1.5 Pro

**コストが低いモデル（高速・軽量）**:
- Claude 3 Haiku
- GPT-3.5 Turbo
- Grok 4

### コスト最適化のヒント

#### 1. モデルの使い分け
```
✅ 軽いタスク → 低コストモデル（Claude Haiku、GPT-3.5）
✅ 重要なタスク → 高性能モデル（Claude Opus、GPT-4.1）
✅ 大規模分析 → Gemini 2.5 Pro（必要な時だけ）
```

#### 2. コンテキストの最適化
```
❌ プロジェクト全体を毎回読み込む
✅ 必要なファイルだけを選択して読み込む
✅ 関連するファイルに絞ってコンテキストを提供
```

#### 3. 機能の使い分け
- **タブ補完**: 低コストモデルで十分（自動的に最適化される）
- **Chat Mode**: 標準的なモデル（Claude 3.5 Sonnet、GPT-4.1）
- **Composer Mode**: 必要に応じて高性能モデル
- **大規模リファクタリング**: 重要度が高い場合のみ最上位モデル

#### 4. 使用量の監視
- Cursorダッシュボードで使用状況を定期的に確認
- モデル別の使用量とコストを把握
- 月の途中で使用量が多い場合は、モデル選択を見直す

### 超過料金について

- 月額プランに含まれる使用量を超過した場合、従量課金が発生
- 超過分は各モデルのAPIコストに基づいて計算
- ダッシュボードで現在の使用量と残高を確認可能
- 予算管理のために、使用量アラートの設定を推奨

### プラン選択の決め方

**質問1**: 毎日どのくらいAIを使いますか？
- 時々コード補完だけ → **Pro**
- 毎日数回のコード生成 → **Pro**
- 毎日積極的にAI活用 → **Pro Plus**
- 常時AIと共同開発 → **Ultra**

**質問2**: 主にどの機能を使いますか？
- タブ補完中心 → **Pro**
- Chat + Composer → **Pro Plus**
- 大規模プロジェクト分析 → **Ultra**

**質問3**: チームで使いますか？
- 個人利用 → **Pro/Pro Plus/Ultra**
- チーム利用 → **Teams**
- 企業全体 → **Enterprise**

## 9. 注意事項

- **コンテキスト制限**: 各モデルにはトークン制限があるため、大規模なプロジェクトでは適切にファイルを選択する必要があります
- **コスト管理**: 高性能モデルは使用料が高いため、タスクの重要度に応じて使い分けましょう
- **最新情報**: モデルは定期的にアップデートされるため、Cursor公式の最新情報を確認してください
- **プライバシー**: 機密情報を含むコードを扱う際は、各モデルのデータ保持ポリシーを確認してください
